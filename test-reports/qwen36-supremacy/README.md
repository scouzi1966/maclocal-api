# Qwen3.6-27B Local-Engine Supremacy Benchmark

Compares **afm** against other Apple-Silicon LLM engines on **Qwen3.6-27B (4-bit)**:
mlx-vlm (official MLX Python server), LM Studio, oMLX, Rapid-MLX, llama.cpp (GGUF), ollama.

- **`SUPREMACY-REPORT.md`** — the written analysis + verdict.
- **`results/benchmark-data.{json,csv}`** — consolidated, plot-ready dataset (regenerated by `aggregate.py`).
- **`results/*`** — raw per-engine artifacts (probe JSON, llama-benchy MD tables, server logs).
- **`scripts/*`** — the harness.

## Quick start

```bash
cd test-reports/qwen36-supremacy/scripts

# one-time: isolated mlx-vlm env (only needed for run-mlx_vlm.sh)
./setup-vlmenv.sh && export VLM_PYTHON=/tmp/bench/vlmenv/bin/python

# run one engine (kills any other engine first, writes to ../results/):
./run-afm.sh
./run-llamacpp.sh

# run the whole matrix sequentially, then rebuild the dataset:
./run-all.sh

# rebuild benchmark-data.{json,csv} from whatever is in ../results/:
./aggregate.py
```

## Output location

Results default **into this repo** at `test-reports/qwen36-supremacy/results/`.
Override per run with `BENCH_OUT_DIR=/some/path ./run-afm.sh`.

## Methodology notes (why the dataset is trustworthy)

- **Canonical metric = `probe.py`, one uniform formula for every engine:**
  `decode tok/s = server usage.completion_tokens ÷ decode_window` (window = first→last
  streamed-delta time). The token count is the server's own standard `usage` field, so it's
  immune to client-side re-tokenization *and* to chunk-batching. `aggregate.py` computes this
  identically for all engines (llama.cpp carries a server-`eval time` cross-check).
- **Why not llama-benchy / guidellm?** Both proved unreliable for these local servers.
  llama-benchy's `tg` is inflated for servers without streamed `token_ids` (afm ~29, oMLX
  "peak 156" — impossible for a 27B; it re-tokenizes, sometimes with a gpt2 fallback).
  guidellm 0.6.0 counts tokens correctly but failed to capture TTFT/ITL for afm **and** the
  reference `llama-server` (and returned nonsense for oMLX, failed on ollama). No engine here
  streams `token_ids`, so the server-`usage`-anchored probe is the only consistent basis.
- **One engine on the GPU at a time** (each runner kills the others first); memory watchdog
  aborts if free RAM < `MIN_FREE_GB`.

## Engine availability for Qwen3.6 (`qwen35`)

| Engine | Loads? | Note |
|---|---|---|
| afm, mlx-vlm, LM Studio, oMLX | ✅ | MLX 4bit |

| ollama 0.24.0 — MLX backend | ✅ | `qwen3.6:27b-mlx` (NVFP4, ~19 GB) — `OLLAMA_MODEL_ID=qwen3.6:27b-mlx ./run-ollama.sh` |

| `mlx_lm.server` (text-only) | ❌ | VLM checkpoint → 333 params not in model |
